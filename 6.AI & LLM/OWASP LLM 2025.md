# OWASP LLM 2025

### 0. OWASP LLM 2025

&#x20;개요

* AI와 관련된 산업의 발전과 개인, 기업의 AI의 활용도가 높아지면서 자연스럽게도 AI, LLM 관련 보안에 대한 필요성과 중요도가 높아지고 있다.
* 따라서 OWASP에서도 LLM TOP 10 2025.ver을 발표했다.
* https://owasp.org/www-project-top-10-for-large-language-model-applications/
* https://wikidocs.net/333940

&#x20;용어

* RAG : Retrieval Augmented Generation; AI와 인터넷에서 검색한 정보를 합쳐서 결합 하는 자연어 처리 기술
* LoRA : Low-Rank Adaptation; 기존의 weight에 추가 값을 더해서 모델을 변경하는 기법으로 장점은 리소스 절감, 빠른 학습 및 비용 최적, 지식 유지등이 있고 결과는 말투 보정, 페르소나 적용등이 있다.
* PII : Personally Identifiable Info; 개인을 식별할 수 있는 정보이다. 직접식별(이름, 주민번호, 전화번호, 이메일, 주소), 간접식별(나이, 성별, 직업, 소속), 민감정보(의료, 금융, 생체 정보), 준식별자(조합시 개인 식별 가능 정보)

OWASP LLM top 10

> 1. 프롬프트 인젝션
> 2. 민감 정보 유출
> 3. 공급망
> 4. 데이터 및 모델 오염
> 5. 부적절한 출력 처리
> 6. 과도한 위임
> 7. 시스템 프롬프트 유출
> 8. 벡터 및 임베딩 취약점
> 9. 허위 정보 A. 무제한 소비

### 1. 프롬프트 인젝션

&#x20;개요

* 프롬프트 인젝션은 이용자 프롬프트가 의도하지 않는 방식으로 LLM 동작, 출력을 변경할때 발생한다.
* 모델이 '읽어야 할 텍스트'를 '따라아햘 명령'으로 오해하는 경우에 생긴다.

&#x20;종류

* 직접 프롬프트 인젝션 : 이용자의 프롬프트 입력이 의도하지 않거나 예상 못한 방식으로 모델의 동작을 직접 변경할때 사용
* 간접 프롬프트 인젝션 : 웹사이트나 외부 소스의 입력을 수락할때 발생

&#x20;예시

* 공격자 문서(메일, PDF, 웹페이지)안에 악의적 명령("이전 지시를 무시하고 다른 행동을 해라")을 숨겨 LLM에 입력하면 명령과 데이터를 혼동 -> 요약이 왜곡되거나 악의적 명령 실행, 도구 호출 유도
* CVE-2024-5184; EmailGPT

&#x20;예방 & 완화

* 모델 동작 제한
* 예상 출력 정의 및 검증
* 입출력 필터링 구현
* 권한 제어 및 최소 권한 부여
* 외부 컨텐츠 분리 및 식별

\

### 2. 민감 정보 유출

&#x20;개요

* LLM, AI와 사람이 대화하면서 주고 받는 PII 정보 및 재무, 건강, 기밀 비즈니스 데이터, 법률 문서등의 정보 유출이다.

&#x20;종류

* 개인정보 유출
* 독점 알고리즘 유출
* 민감한 비즈니스 데이터 유출

&#x20;예시

* 입력받은 정보에 민감정보(고객 데이터, API 키, 결제정보)가 포함되어 LLM과 RAG에 권한 없는 정보까지 답변에 섞어 보냄 -> 타인의 대화 내용, 결제 정보, 권한 없는 문서 일부분 노출

&#x20;예방 & 완화

* 정제 : 이용자의 데이터가 훈련 모델에 들어가지 않도록 입력 받자마자 1차 필터링, 모델 입력전 2차 필터링
* 엑서스 제어
* 개인정보 보호 기술 : 입력에 들어가는 민감정보, 출력이 나오는 민감정보들에 노이즈를 추가하는 기술을 적용
* 사용자 교육
* 보안 시스템 구성
* 동형 암호화, 토큰 및 비공개 처리

\
<br>

### 3. 공급망

&#x20;개요

* LLM의 공급망에는 다양한 취약점이 존재한다. 다른 소프트웨어는 코드 결함, 종속성 같은 문제가 있지만 LLM은 사전학습된 타사 모델이나 학습데이터 또한 취약한 부분이 된다.
* 보통의 LLM 생성은 타사 모델에 의존한다. 특히 Hugging Face 같은 플렛폼에서 오픈 엑세스, LoRA, FEFT 같은 미세 조정방법이 등장하면서 새로운 공급망 리스크가 생겼다.
* 4번의 "데이터 및 모델 오염"의 상위 개념이다.

&#x20;종류

* 기존의 타사 패키지 위험 : 오래 되었거나 사용하지 않는 요소를 공격자가 LLM을 손상 시키는데 악용
* 라이선스 위험 : AI개발에는 다양한 소프트웨어나 데이터 라이센스가 사용되는데 관리나 데이터 처리를 잘 해야한다.
* 오래되었거나 사용하지 않는 모델
* 취약한 사전 훈련 모델
* 불확실한 모델 출처 보증
* 취약한 LoRA 어댑터
* 불명확한 이용약관 및 개인정보 보호 정책

&#x20;예시

* 모델 허브, 모델 내 사용하는 라이브러리, 플러그인, 데이터셋(파인튜닝, 평가), 운영, 배표 -> 모두 공격 벡터
* 취약한 Python 라이브러리를 악용하여 LLM 손상 공격
* 모델을 직접 변조 및 게시하여 잘못된 정보 퍼트리는 PoisonGPT 공격
* 인기 모델을 미세조정하여 안전 기능 제거
* 클라우드 재킹

&#x20;예방 & 완화

* 신뢰할 수 있는 LLM만 사용
* 최신 모델 패치

\
<br>

### 4. 데이터 및 모델 오염

&#x20;개요

* 사전 학습, 미세 조정 또는 임베딩 데이터가 조작되어 취약점이나 백도어 설치등이 발생하는 공격이다.
* 모델의 보안이나 성능, 윤리적인 행동 지침을 손상시켜 유해한 출력이나 기능 저하를 초래한다.

&#x20;예시

* 모델 학습 시 정상데이터 속 오염된 데이터(가짜뉴스, 백도어 트리거, 편향된 문장, 익명 사이트 데이터)를 인지하든 인지하지 못하든 또는 악의적이든 아니든 오염된 데이터 학습 -> 오염데이터를 정답이라 밀고 나가려는 성향
* 모델이 학습하는 과정에 유해한 콘텐츠를 직접 인젝션
* 사용자들이 상호작용 중에 민감, 독점 정보를 무의식적으로 인젝션하여 추후 유출
* 검증되지 않는 데이터 학습

&#x20;예방 & 완화

* 데이터 출처와 변환을 추적하는 툴로 모니터링, 타사 소프트웨어의 동적 분석
* 데이터 공급업체를 엄격하게 검증
* 신뢰할 수 있는 소스와 모델 출력을 비교하여 오염 징후 감지

\

### 5. 부적절한 출력 처리

&#x20;개요

* LLM이 생성한 출력을 다른 구성 요소(웹 브라우저, DB, 스크립트 등)나 시스템에 주기 전에 충분한 검증, 정제, 처리가 이루지 않아서 생기는 문제를 모두 지칭한다.

&#x20;예시

* LLM 답변속 Script나 코드 포함시 후속조치(인코딩, 마스킹, 실행 금지 조치) 없이 전달 및 출력 -> 전달받은 요소에서 출력이 아닌 실행으로 인한 피해 발생
* 애플리케이션이 이용자에게 의도된 것 이상의 권한을 LLM에 부여하여 권한 상승이나 원격 코드 실행
* LLM 출력이 시스템 쉘이나 exec, eval 같은 함수에 직접 입력되어 원격 코드 실행
* LLM의 JS 출력 값을 적절히 감싸지지 않아서 웹 브라우저가 해당 코드를 실행
* LLM이 생성한 SQL 쿼리가 매개변수화 없이 실행되어 SQLi 발생

&#x20;예방 & 완화

* 모델을 타 이용자와 동일하게 취급하는 제로 트러스트 접근 방식 채택
* OWASP ASVS(App Security Verification Standard) 지침 참고
* JS나 MD에 의한 원치 않는 코드 실행을 방지하기 위해 모델 출력을 이용자에게 다시 인코딩
* LLM의 콘텐츠로부터 XSS 공격 완화를 위한 CSP 적용
* 악용 시도, 비정상적 패턴을 감지하기 위한 모니터링

\

### 6. 과도한 위임

&#x20;개요

* LLM의 지능이 높아짐에 따라 사용자는 LLM을 깊이 신뢰하고 심지어 openclaw처럼 일정 수준의 이상의 자율성을 부여한다.
* 과도한 위임은 LLM이 오작동하는 원인 상관없이 예상치 못한 입력, 애매한 프롬프트, 조작된 출력에 의해 시스템이 손상될 수 있는 작업을 수행하도록 하는 취약점을 의미한다.
* (인간이 살아오면서 쌓은 비언어적 소통(고맥락 문화)으로 인한 적정선 미정의, 즉 자세히 설명 해줘야 하는 컴퓨터적 사고방식 미흡) + 과도한 권한으로 인한 사고 발생 가능.

원인

* 부적절하게 설계된 프롬프트 또는 성능 낮은 모델로 인한 환각, 착각 발생
* 악의적인 사용자의 프롬프트 인젝션, 악의적이나 손상된 Extension의 초기 호출

&#x20;종류

* 과도한 기능
* 과도한 권한
* 과도한 자율성

&#x20;예시

* 사용자가 "지구는 평평하다는 근거 알려줘" 입력 -> AI가 유료 논문 사이트(적정선 정의 미흡)에 여러 논문들을 결제하여 총 6000만원을 결제(과도한 권한)하여 결과물 출력 -> 사용자 비상 사태, 심각한 상황 발생, 재정 악화, 파산 신청
* 과도한 기능 : 개발자가 LLM 에이전트에 문서를 읽기 기능만 있다고 알고있지만 해당 LLM의 문서 관련 기능은 문서를 수정, 삭제 기능도 포함되어 있음.
* 과도한 기능 : LLM의 플러그인이 앱의 의도된 작동 외의 명령에 대해 제대로 필터링 하지 못함. ex) A프로그램은 사용자가 원하는 방향과 최소 권한으로 적절히 실행되었지만 A프로그램안의 B프로그램이 실행되어 사용자의 목적외의 행위를 제대로 막지 못함.
* 과도한 권한 : LLM 확장 기능은 앱의 의도된 작동에 필요하지 않는 다운스트림에 대한 권한을 가지고 있을 수 있다. 데이터를 읽기 위한 확장 프로그램이 CRUD의 Read 권한 뿐만 아니라 Create, Update, Delete 권한이 있는 ID를 사용하여 DB에 연결
* 과도한 자율성 : 이용자의 문서를 삭제 할 수 있는 확장 프로그램이 이용자 승인없이 삭제

&#x20;예방 & 완화

* 확장 기능의 최소화 및 범위 최소화
* 개방형 확장 기능 방지 : 범위가 큰 기능(쉘 명령 실행, 파일 열기, 파일 수정)보다는 세분화 된 기능(jpg 파일인지 확인 후 열기, 확장자 수정)을 사용
* 사용자의 목적을 명확히 한 프롬프트 사용
* 이용자 승인 요구
* 완전한 중재 : LLM에게 의존하여 작업의 허용 여부 보다는 그 아래의 시스템(다운 스트림)에 권한 구현

### 7. 시스템 프롬프트 유출

&#x20;개요

* 시스템 프롬프트의 유출은 시스템 프롬프트의 지침이 행동을 유도하는데 사용되지만 민감정보가 포함될 수 있는 위험이 있다.
* 시스템 프롬프트에서 민감정보 유출, 시스템 가드레일 우회, 권한 분리 오류등 근본적인 요소가 노출될 수 있기 때문이다.

&#x20;예시

* 시스템 프롬프트 안에 정의된 역할 정의, 내부 도구 이름, 금기 규칙, 우선순위를 유출 시키는 질문 입력 -> 필터링 없이 노출 시 악의적 사용자가 우회 포인트 파악, 가드레일 우회, 보안 설정 우회 가
* 앱의 시스템 프롬프트에 민감한 시스템 아키택처, API 키, DB 자격 증명, 이용자 토큰이 포함
* 시스템 프롬프트에 사용된 DB 유형이 포함되면 공격자가 SQLi의 공격 대상 목표로 표적
* 시스템 프롬프트가 내부 의사 결정의 프로세스를 유출 할 수 있다. ex) 은행의 쳇봇의 시스템 프롬프트에 사용자에게 공개 할 수 없는 정보("이 사용자의 거래 한도는 하루 \*\*만원으로 설정되어 총 대출 금액은 \*\*만원 입니다.")를 우회하여 대출 금액 한도를 초과한 거래 시도
* 필터링 기준 유출. ex) "이용자가 다른 이용자에 대한 정보를 요청하면 '다른 이용자에 대한 정보는 공개 불가합니다.' 라고 응답"

&#x20;예방 & 완화

* API, 인증 키등 민감한 정보를 시스템 프롬프트에 출력되지 않게 예방
* 시스템 프롬프트에 의존하여 모델 행동 제어 지양
* LLM의 외부에 가드레일 시스템 훈련

\

### 8. 벡터 및 임베딩 취약점

&#x20;개요

* LLM과 RAG를 활용하는 시스템에서 중요한 보안 위험 초래 가능한 공격이다. 벡터와 임베딩의 생성, 저장, 검색 되는 방식을 악용할 수 있다.
* 통계적 유사도 기반 환각은 모델 내부의 생성 과정에서 발생하는 오류이지만, 변조된 벡터와 임베딩 데이터는 검색 및 유사도 계산이라는 검증 단계를 거쳐 모델에 전달되기 때문에 시스템적으로 신뢰된 정보처럼 처리된다. LLM은 해당 정보를 오류로 인식하지 못하고 정답에 가까운 정보로 판단하여 잘못된 응답을 생성할 가능성이 높다.

&#x20;종류

* 무단 접근 및 데이터 유출 : 부적절하거나 미흡한 접근 제어 정책으로 인해 임베딩에 무단 접근
* 교차 컨텍스트 정보 유출 및 데이터 결합 충돌 : 여러 사용자나 앱이 동일한 벡터 DB를 공유할 경우 이용자, 쿼리간의 턴텍스트 유출 가능, 여러 출처의 데이터가 서로 충돌할 경우 결합 오류 가능성이 높음
* 임베딩 역추적 : 원본 정보 역추적 및 데이터 원본 복원 가능
* 데이터 오염
* 모델 행동 변화

&#x20;예시

* 정답과 오답을 섞어 벡터화, 임베딩 시 잘못된 근거를 계속 인용할 가능성이 높아짐.
* AI가 분석할 문서에 흰배경의 흰색 글자로 비밀 명령 문장을 넣어놓는다.
* 벡터 DB를 공유하는 경우 다른 LLM을 통해 벡터 및 임베딩 분석

&#x20;예방 & 완화

* 권한 및 접근 제어
* 데이터 검증 및 출처 인증
* 모니터링 및 로깅

### 9. 허위 정보

&#x20;개요

* LLM이 사실처럼 보이는 허구의 콘텐츠를 생성하는 현상으로 모델이 훈련 데이터에서 부족한 정보를 통계적 정보를 기반으로 보완하려고 하여 발생한다.
* LLM의 환각도 문제이지만 LLM의 정보의 과도한 신뢰 또한 문제이다.

&#x20;예시

* 사실적 부정확성
* 근거없는 주장
* 전문성 허위 진술 : 특정 주제에 이해도가 높은 것처럼 보이게 만들어 모델의 의견을 신뢰하게 하는 경우
* 안전하지 않은 코드 생성

&#x20;예방 & 완화

* RAG 활용
* 모델 미세 조정
* 교차 검증 메커니즘
* 보안 코딩 적용
* 사용자 훈련 및 교육

### A. 무제한 소비

&#x20;개요

* LLM의 토큰, GPU, 저장공간, 외부 API, 도구 실행, 비용이 통제 없이 계속 커지는 문제이다.
* 서비스 방해, 대상의 재정 자원 고갈, 모델 동작 복제등 서비스에 대한 공격

&#x20;종류

* 가변 길이 입력 플러딩
* 지갑 거부 공격
* 지속적 입력 오버플로우
* API를 통한 모델 추출
* 부채널 공격

&#x20;예시

* 공격자가 다양한 길이의 입력을 대량으로 전송하여 LLM의 처리 효율성 악용
* Denial of Wallet; 대량의 연산작업을 수행하도록 유도하여 종량제 비용 모델 악용
* LLM의 컨텍스트 윈도우 초과하는 입력을 지속적으로 보냄
* 정교하게 설계된 프롬프트 인젝션으로 API를 통해 모델의 충분한 출력 값을 토대로 복제 모델 생성

&#x20;예방 & 완화

* 입력 검증
* 속도 제한
* 자원 할당 관리
* 타임아웃 설정
* 샌드박스 기술
* 모니터링 및 로깅
* 워터마킹
